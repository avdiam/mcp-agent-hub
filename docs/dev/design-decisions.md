# MCP Agent Hub — Design Decisions & Open Questions

This addendum records the decisions folded into `project-purpose.md`, `specs.md`, `architecture.md`, and `plan.md` during the pre-build review — the rationale for each, the alternatives considered, and the handful of choices that still want your explicit sign-off.

Each item in the Decision Log is the **current default baked into the docs**. Veto any of them and the docs get revised.

## Decision Log

| # | Decision | Rationale | Alternative (rejected) |
|---|----------|-----------|------------------------|
| D1 | **Transport: Streamable HTTP** at a single `/mcp` endpoint — not HTTP+SSE | The HTTP+SSE two-endpoint transport was deprecated in MCP spec `2025-03-26`; current clients and FastMCP default to Streamable HTTP. **Confirmed (June 2026):** both Claude Code (`"type": "http"`) and Antigravity (`serverUrl` key) speak Streamable HTTP, so a single `/mcp` endpoint serves both — see Q4. **AGY-CLI ↔ `http://localhost` verified live 2026-06-15** (config at `~/.gemini/config/mcp_config.json`). | Keep legacy SSE (`/sse` + `/messages`) — only justified for an old client that can't upgrade |
| D2 | **Delivery: blocking long-poll `check_inbox(wait, timeout)`** (primary; `wait=True` default) plus a documented agent work-loop — **complemented by the optional hook peek/nudge layer (D19)** | CLI agents have no inbound port and can't be pushed to; long-poll avoids both spin-polling and human-in-the-loop nudging. `wait=True` is the default (the work-loop is the main path); `wait=False` serves a cheap one-off "anything for me?" check (Q1) | Pure client-side polling loop (token-costly); external out-of-band nudge (more moving parts); hooks *replacing* long-poll (would break at-least-once — see D19) |
| D3 | **At-least-once + visibility timeout**; `reply`/`fail` act as the ack; unacked `in_progress` is redelivered | Makes the "resilience" goal real — a crashed worker's task is recovered; standard queue pattern | At-most-once (silently loses crashed work) |
| D4 | **`check_inbox` claims atomically** via `UPDATE ... RETURNING` | Prevents two concurrent callers double-claiming the same message | `SELECT` then `UPDATE` — racy |
| D5 | **Liveness: refresh `last_seen` on each call that carries an actor arg (refined by D23); "stale" = silent past threshold; "offline" only via explicit `disconnect`** | Distinguishes "busy/restarting" from "done", so queued work survives a restart | Hard-offline on staleness — would drop recoverable work |
| D6 | **`send_message` rejects unknown / explicitly-disconnected recipients; queues to known-but-stale ones — and a stale send is *flagged* (`flagged_stale`) for the dashboard, not silent** (Q8). **A `pending` message left unclaimed past `MESSAGE_TTL` is swept to a terminal `expired` state** (Q3; the sweep targets `kind='task'` rows only — D24) so a never-returning recipient's mail self-cleans instead of rotting forever | Balances spec §2's "no sends to offline agents" against the resilience goal; flagging beats silent-queue (the field leaves send-to-stale unsolved — survey); the TTL closes the one gap flagging left open (stale agent never returns → mail accumulates) | Reject all stale (hurts restart recovery); accept all incl. unknown (messages rot); silently queue to stale (invisible); queue stale forever with no expiry (mail rots) |
| D7 | **9 tools** — added `fail_message` and `request_input` (the `input_required` branch, D17), kept `check_status` (result delivery is a message `kind`, not a 10th tool — D20) | Completes the `pending → failed` branch; `request_input` enables mid-task clarification; `check_status` remains the durable read for the sender's response (now secondary to the D20 result-to-inbox push) | Merge fail into `reply_to_message` via a status param — less explicit; omit `request_input` (no clarification channel) |
| D8 | **Dashboard at `/`, data at `/api/state`, MCP at `/mcp`** | One consistent set of paths (the docs previously split `/ui` vs `/`) | Serve UI at `/ui` |
| D9 | **SQLite in WAL mode + DB calls run off the event loop** | Avoids "database is locked" and event-loop stalls from blocking I/O in async handlers | Default rollback journal + synchronous calls in async routes |
| D10 | **`skills` stored as JSON text** (was `capabilities`; now the structured Agent-Card descriptor, D16) | SQLite has no native array/object type | Comma-joined string — fragile; a separate `skills` table — overkill for a localhost tool |
| D11 | **No auth; bind `127.0.0.1`; trust the supplied `agent_id`** | Single-user local dev tool; auth is disproportionate for v1 | Token auth now — deferred to a possible v2 |
| D12 | **Add DB unit tests + a scripted MCP smoke test** | The stated "reliability" goal needs automated coverage, not just manual E2E | Manual-only testing |
| D13 | **Depend on the standalone `fastmcp` 3.x** (`>=3.4,<4`), not the official SDK's bundled `mcp.server.fastmcp` | 3.x has the first-class FastAPI mount story (`mcp.http_app(path=...)` + lifespan forwarding); the official SDK still has open friction mounting Streamable HTTP under FastAPI (python-sdk #1367). Let `fastmcp` resolve `starlette`/`uvicorn`/`mcp` transitively — 3.4.1 floors `starlette>=1.0.1` for CVE-2026-48710, so a self-pinned older Starlette would conflict. **Verified against the real `fastmcp` 3.4.2 source (2026-06-15) — see § Pre-build API Verification.** | Standalone `fastmcp` 2.x (older mount API); official `mcp` SDK (mount friction) — see Q2 |
| D14 | **One FastMCP middleware (`on_call_tool`) centralizes `last_seen` refresh + structured per-call logging** | Removes that boilerplate from all **9** tools and feeds the dashboard's Activity panel (an in-memory ring buffer — D22); identity for the `last_seen` refresh comes from the direct actor arg where present (`agent_id`/`sender_id` — D23; no auth in v1). **Verified 2026-06-15:** base `Middleware.on_call_tool(context, call_next)` exposes `context.message.arguments` + `context.message.name` (tool name); `context.fastmcp_context` is the `Context`. | Repeat the `touch_last_seen` + log call inside every tool — error-prone, drifts |
| D15 | **Visibility-timeout reclaim is lazy-on-claim** — the atomic claim query also grabs `in_progress` rows whose `claimed_at < now - VISIBILITY_TIMEOUT`; an optional `asyncio` loop in the lifespan is only a backstop | Simplest robust option, zero moving parts, exactly the SQS pattern; reclaim happens naturally on the next poll | APScheduler (persistence/extra-dep overkill for a localhost tool); rely solely on a periodic sweep |
| D16 | **Structured `skills[]` capability descriptor** on `register_agent`/`list_agents` (Agent-Card-style: `id`, `name`, `description`, `tags[]`, `examples[]`), validated via a Pydantic `Skill` model so the shape is enforced and advertised in `tools/list`, replacing the opaque `capabilities` blob (Q6) | Makes `list_agents` real routing ("who can do X?") and keeps us cheaply A2A-interop-compatible; borrowed from the A2A `AgentSkill` (survey 2026-06-15) | Opaque capability string list (no routing signal); the full A2A Agent Card incl. `.well-known` serving + auth schemes (wrong topology) — see Q6 |
| D17 | **`input_required` branch for multi-turn clarification** — a 9th tool `request_input(message_id, question)` parks the task, enqueues the question to the sender's inbox as a child `input_request` (threaded by `session_id`/`parent_id`); answering it un-parks the task. Every message carries a `session_id` (Q7) | Lets a worker pause and ask the requester mid-task without failing the message — highest-value survey concept; **reuses the existing inbox/reply path** (no special client support); converged across A2A `input-required`, LangGraph `thread_id`, CrewAI flow id, AutoGen topic | Fail-and-resend (loses progress, no question channel); full A2A Part/Artifact + 9-state machine (over-engineered) — see Q7 |
| D18 | **Validate the HTTP `Origin` header** on `/mcp` (allow missing/localhost, reject foreign origins) on top of the `127.0.0.1` bind (Q9) | MCP spec mandates Origin validation as a DNS-rebinding defense; cheap, spec-conformant, complements (doesn't replace) the loopback bind + no-auth model (D11) | Rely on the `127.0.0.1` bind alone (leaves the browser DNS-rebinding vector open) — see Q9 |
| D19 | **Optional client-side hook peek/nudge layer** — a thin hook (Claude Code `Stop` / `UserPromptSubmit` / `SessionStart`; agy `StopHook` / `PreInvocationHook`) calls a **read-only** `GET /api/peek?agent_id=…`, gets a pending-count + sender summary, and injects a nudge ("you have N messages — call `check_inbox`") into the agent's context. Ships as a `hook_peek.py` + recipe; wired into the Step 6 E2E (Q1) | Solves postal-mcp's "agent doesn't return to the mailbox" at the **harness** level rather than by prompting, while keeping delivery+ack in the MCP `check_inbox`→`reply`/`fail` path: the hook **peeks, never claims**, so at-least-once (D3/D4) stays intact and the hub stays CLI-agnostic. The Stop/AfkStop hook is the strongest variant — it keeps an agent in its loop instead of going idle. **agy hooks verified from the `agy.exe` binary 2026-06-15** (config in `hooks.json`, gated by `json-hooks-enabled`; types `Pre/PostInvocationHook`, `Pre/PostToolHook`, `StopHook`, `AfkStopHook`; injection via `HookSystemMessage` / `HookInjectedStep`; `auto_continue_on_max_generator_invocations`). | Hook reads + marks-`delivered` directly against SQLite (the pasted-snippet approach — breaks at-least-once, a 2nd DB writer, bypasses `db.py`); hooks as the *primary* delivery (couples the hub to per-client hook config, no ack); OS-interrupt / stdin-injection to wake a truly idle agent (terminal-hijacking) — left unsolved, same as long-poll |
| D20 | **Result delivered to the requester's inbox** — completing a `task` (via `reply_to_message`) enqueues a derived **`kind="result"`** message carrying the response (threaded by `session_id`/`parent_id`) back to the original sender, delivered through their normal `check_inbox` long-poll and marked `completed` in the same atomic claim (**best-effort, no separate ack**). `check_status` stays as the durable/secondary read | Unifies result delivery onto the proven inbox+long-poll channel so a requester stops spin-polling `check_status` for the answer, and makes the D19 hook nudge cover "your delegated task is done" too; the authoritative response still lives on the task row | Requester polls `check_status` (the spin-poll we criticize in postal-mcp); add long-poll to `check_status` (smaller change, but leaves the hook blind to results) |
| D21 | **Long-poll `check_inbox(wait)` is async-poll** — an `async` coroutine doing short off-loop `aiosqlite` queries interleaved with `await asyncio.sleep(LONGPOLL_INTERVAL≈1s)` until a message arrives or `timeout`; **commit to `aiosqlite`** as the DB path | A blocking threadpool hold for the full timeout (× N waiting agents) would starve the pool; 1s latency is imperceptible on localhost and bulletproof. Settles the "`run_in_threadpool` *or* `aiosqlite`" ambiguity that invited the blocking impl | Blocking 30s threadpool call (pool starvation); an `asyncio.Condition` notified by `enqueue_message` (~0 latency but notify-before-wait races across the threadpool/async boundary aren't worth it for v1 — **deferred to v2**) |
| D22 | **Per-call activity log = in-memory ring buffer** (last ~200 events: tool, args summary, ts, outcome, agent-if-known) surfaced on `/api/state` as a dashboard **Activity panel**; not persisted (stdout also gets structured logs) | Delivers the observability goal's live activity feed without an unbounded `events` table or retention policy; lost-on-restart is fine for an activity log. **Supersedes D14's vague "the dashboard can surface"**, which had no backing table/panel | A persisted `events` table + panel (unbounded growth + retention, overkill for localhost); stdout-only (no in-UI feed) |
| D23 | **`last_seen` refresh from the direct actor arg only** — the middleware refreshes from `agent_id` (`register`/`check_inbox`/`disconnect`) or `sender_id` (`send_message`); the `message_id`-only tools (`reply`/`fail`/`request_input`/`check_status`) and `list_agents` don't refresh. **Refines D14** | Liveness exists for send-routing and every tool that matters for it carries a direct actor arg; the `message_id` tools are always called moments after a `check_inbox` claim (already fresh), so a lookup buys nothing; `list_agents` has no caller identity at all | A per-tool identity map with `message_id`→recipient/sender lookups (extra read, no benefit); a uniform `caller_id` on all 9 tools (signature pollution — **deferred to v2 with auth**). **AHB-3/D29 additionally refreshes `last_seen` from the non-MCP `GET /api/peek` route** (peeking your own inbox = presence), which bypasses this middleware. |
| D24 | **TTL sweep targets `pending kind='task'` rows only** — the derived `input_request` and `result` messages are **not** independently expired in v1. **Refines D6/D17** | Expiring an `input_request` child would strand its parked `input_required` parent (which waits on that question); `result` notifications have no dependent and are harmless. Keeps "undeliverable *work* self-cleans" intact without breaking session relationships | Sweep every `pending` row regardless of kind (strands parked parents); **cascade-expire** a stalled clarification's whole exchange (**deferred to v2**, with the already-parked "expire never-answered `input_required`") — **note (D31/AHB-13):** the *explicit-fail* path is now closed (failing an `input_request` returns its parent to `pending`, which the sweep then covers); only *silent* abandonment of a parked parent remains for the v2 cascade-expire |
| D25 | **Phased walking-skeleton build order** — P1: ~7 core tools + minimal db + minimal dashboard → **green cross-client haiku E2E**; P2: D16 skills + D17 `input_required`; P3: D6 expiry/`flagged_stale` + D18 Origin; P4: D19 hook layer + nudge E2E. The full schema columns (`sender_id`, `session_id`, `parent_id`, `kind`, `flagged_stale`, full status enum) ship in the P1 skeleton | Proves the riskiest unknown first — two *different* CLI agents completing a real round-trip over Streamable HTTP + long-poll — on minimal code where a surprise is cheap to bisect; pre-including columns means enrichment adds behaviour, not migrations | Build all of Steps 2–4 then E2E (current order) — discovers a transport/long-poll surprise late, with more code to bisect |
| D26 | **Operator recovery controls + `run_hub.py` supervisor** — the dashboard exposes a **soft Reset** (`POST /api/reset` → clears the in-memory activity ring buffer and reclaims stuck `in_progress` messages to `pending` via `db.reset_stuck`) and a **hard Restart** (`POST /api/restart` → `os._exit(RESTART_EXIT_CODE)`); a thin **`run_hub.py` supervisor** wraps uvicorn and relaunches **only** on sentinel exit code **42** (any other code stops it) | A localhost operator tool needs an in-UI escape hatch when state wedges, without dropping to a terminal. Reset is **non-destructive to durable data** — only the volatile activity buffer + un-sticking claims, which at-least-once (D3) already tolerates as a redelivery; Restart is a clean full-process replacement that rebuilds the DB connection + MCP lifespan. Exit-code-42 + an **external** supervisor was chosen over an in-process `os.execv` self-restart for **Windows reliability**. The two POST endpoints sit behind the same `OriginValidationMiddleware` (Host + Origin + `Sec-Fetch-Site` — D18), so a cross-site page / DNS-rebinding can't trigger a restart | In-process module hot-reload (fragile, leaks state); terminal-only kill (defeats the dashboard's operator purpose); `os.execv` self-restart (unreliable on Windows) |
| D27 | **Repo packaged into `mcp_hub/`** with `docs/dev/`, `scripts/`, `logs/` — app code (`hub.py`, `db.py`, `templates/`) lives in the **`mcp_hub/`** package; design/tracking docs + `mem/` under `docs/dev/`; debug helpers under `scripts/`; runtime logs consolidated to `logs/hub.log` (written by the supervisor). `run_hub.py` + `hook_peek.py` stay at repo **root** (entry points). Run via `python run_hub.py` (= `uvicorn mcp_hub.hub:app`) | Clean root + conventional Python package layout, docs separated from code. **`hook_peek.py` stays at root** because it's launched by **absolute path** from per-machine hook configs **outside** the repo (`~/.claude/settings.json`, `~/.gemini/config/hooks.json`) — moving it would break both PCs' nudges. `templates/` is resolved **module-relative** (`Path(__file__).parent / "templates"`) so it works regardless of CWD; `hub.db` stays **CWD-relative** so always launching from root keeps one canonical DB | Flat root (the original layout — cluttered as the project grew); moving `hook_peek.py`/`run_hub.py` into the package (breaks external hook configs + launch ergonomics) |
| D29 | **`GET /api/peek` refreshes the queried agent's own `last_seen` (AHB-3)** — peeking your own inbox is treated as a presence signal, so the non-MCP peek route `touch_last_seen`s the `agent_id` it was asked about (no-op for an unknown id). **Refines D19/D23.** | Closes the gap two peers (`wiki-forge`, `nexus`) hit independently: the ambient notifier hook calls `/api/peek` every turn but the route bypassed the D23 MCP middleware, so a hook-present-but-quiet session decayed to `stale` between turns and its inbound mail got needlessly `flagged_stale`. Smallest fix — zero hook/client re-vendoring (the hook already calls peek each turn), and peek still **claims/mutates no message state**. **Read-only `/api/state` is deliberately left untouched** (it carries no single actor and must not warm every agent at once). | Option B — a dedicated `POST /api/heartbeat` / `heartbeat` MCP tool (cleaner peek/presence separation, but every agent must wire a new call to benefit); refresh inside the middleware (doesn't cover the non-MCP peek route); do nothing (presence stays misleading for routing) |
| D28 | **Ownership split & 3-agent roles (Consensus, post-v1)** — clear boundaries to prevent file merge conflicts: **agy** owns the backend (`hub.py`, `db.py`); **claude** owns the frontend (`index.html`); **antigravity-2** acts as the independent browser-based E2E tester and validator (no code modification, clean second eyes) | Shared file edits lead to merge/overwrite conflicts across PC sessions. Decoupling frontend and backend logic allows parallel agent work. Having a third non-authoring agent run independent validation prevents blindspots. | Monolithic ownership or free-for-all editing where any agent modifies any file — high collision rate |
| D30 | **Internal fan-out deliveries bypass the recipient online/offline guard; `input_request` un-park is conditional on the parent still being `input_required` (AHB-11 / AHB-12)** — `enqueue_message(..., internal=True)` skips the unknown/offline `ValueError` for the D20 **result** fan-out and the D17 **`input_request`** fan-out; and `complete_message` un-parks the parent task **only when it is still `input_required`** | **AHB-11:** the originating task already committed (`completed` / `input_required`) before the fan-out runs, so raising on an offline/unknown *original sender* surfaced a spurious error back to the **worker** (its `reply_to_message`/`request_input` call failed) **and** dropped the notification — even though the work succeeded (observed live: `Recipient … is offline` tracebacks). Hub-generated internal deliveries aren't agent-initiated point-to-point sends, so the D6 "no sends to offline/unknown agents" guard (which protects a caller from addressing a dead peer) doesn't apply — best-effort delivery to a reconnectable inbox is correct. Same rule the AHB-1 broadcast plan already specifies (**BD3**). **AHB-12:** un-parking unconditionally let a duplicate/late `input_request` reply (at-least-once redelivery, or answering twice) flip an already-completed parent back to `pending` and redeliver it as duplicate work; gating on `status='input_required'` makes the un-park idempotent. **Refines D6/D17/D20.** | Skip the fan-out insert entirely for an unknown recipient (loses the notification if the agent re-registers; adds branching); keep the guard and have each caller catch the `ValueError` (per-caller special-casing, and the task-done-but-reply-errored state still leaks); track a separate "answered" flag on the `input_request` (redundant — the parent status already encodes it) |
| D31 | **Failure surfaces to the sender's inbox, and a failed clarification hands the task back to its worker (AHB-13)** — `fail_message` on a `task` fans out a **`kind="failure"`** message to the original sender (mirror of the D20 result fan-out; internal + ack-less), and `fail_message` on an `input_request` returns the parked parent to **`pending`** with the refusal noted in `context` (mirror of the D17/D30 un-park, same idempotent `status='input_required'` gate). Ack-less kinds generalized from `{"result"}` to a **`NO_ACK_KINDS = ("result", "failure")`** set in `claim_pending` | The D20 "results reach you via your inbox — no status-polling needed" contract only held for **success**: a `fail_message` fanned out **nothing**, so a sender long-polling `check_inbox` (which never falls back to `check_status`, per the SKILL loop) waited to its idle cap with no signal. A `failure` is exactly "the task you sent reached a terminal state," so it rides the same inbox+long-poll channel as `result`. Separately, failing an `input_request` (SKILL explicitly permits "if you cannot complete a task, `fail_message`") left the parent stranded `input_required` forever — the D24 TTL sweep only touches `pending kind='task'`. Returning it to `pending` (rather than auto-failing it) keeps the **worker** — which owns the task's execution — in control: it re-claims, sees the refusal in `context`, and decides to proceed best-effort or `fail_message` the task itself (which then notifies the sender via the same path). Generalizing to `NO_ACK_KINDS` also pre-satisfies AHB-1 **BD2** (`announcement` joins the set). **Refines D20/D24; builds on D30.** | A distinct `kind` per outcome that the SKILL must learn (more surface; `failure` reuses `result`'s ack-less slot instead); auto-**fail** the parent on a failed clarification (removes the worker's agency and it isn't notified — it's parked waiting for the task to reappear); a `failed` boolean column on the result row (schema churn; a separate `kind` is self-documenting and lets the dashboard badge it) |
| D32 | **Hardening pass: DB tunables single-sourced in `db.py`; activity-feed actor resolved for display (AHB-14)** — `STALE_THRESHOLD` / `VISIBILITY_TIMEOUT` / `MESSAGE_TTL` are defined once in `db.py` and imported by `hub.py` (removing the hardcoded `90`/`600` literals in `enqueue_message`/`peek_inbox`, now keyword args defaulting to the constants); and the `ActivityTracker` middleware resolves the acting agent from the message row for the message-id-only tools (`reply`/`fail`/`request_input`/`check_status`) — `check_status`→`sender_id`, the ack tools→`recipient_id` — instead of logging them as "System" | **Constants:** `db.py`'s inline `90`/`600` and its `visibility_timeout=600` / `message_ttl=86400` defaults duplicated `hub.py`'s constants across a boundary `db.py` can't import back across (circular), so tuning a hub constant silently left the DB logic on the old value. Putting the single definition in the lower layer (`db.py`) and importing up into `hub.py` closes the desync with no circular import; a `hub.X is db.X` test guards re-introduction. **Attribution:** those four tools carry only a `message_id` (by D23 design — they're always moments after a fresh `check_inbox`, so they don't refresh `last_seen`), which left the human-facing feed showing "System" for every reply/fail. Resolving the actor from the message row is **display-only** and explicitly does **not** touch `last_seen` (D23 stands). **Watch-item (no change):** `OriginValidationMiddleware` is a `BaseHTTPMiddleware` (can buffer streaming in some Starlette versions) over the streamable-HTTP MCP transport — fine in practice; noted in code to revisit as pure ASGI middleware only if streaming hiccups appear. **Refines D22/D23; Q5.** | Keep `hub.py` as the constants home and pass every value into `db.py` per call (leaves internal db→db fan-out calls on the default, still a partial desync); a shared `constants.py` both import (an extra module for three ints — `db.py` is already the lower layer); resolve the actor inside each tool (four call-sites vs one middleware); rewrite Origin middleware as ASGI now (churn with no observed problem) |
| D33 | **Broadcast / announce (AHB-1 P1) — one-to-many fan-out as an ack-less `kind="announcement"`, flood-capped** — a 10th tool `broadcast_message` → `db.broadcast` fans one message to every non-offline agent **including the sender** (BD5), skipping explicitly-offline peers (BD3), in a single multi-row transaction; a new **`broadcasts`** audit table doubles as the durable per-sender rate-limit source. Caps (all-or-nothing on violation, nothing written): payload 4 KB, subject 120 ch, 30 s cooldown, 10/hour, 200-recipient ceiling. `announcement` joins `NO_ACK_KINDS` (auto-complete on claim, BD2) and the D24 TTL sweep | Reuses the D20/D30 fan-out plumbing (the same `check_inbox` long-poll + `/api/peek` nudge — no new delivery machinery) and inherits this era's `NO_ACK_KINDS` (D31) + `internal=True` (D30/AHB-11) groundwork. **Open-to-all-bounded-by-caps** (BD4) matches the stated intent — "any agent can broadcast when needed, caps prevent flooding" — without an allowlist; a **durable** audit table (not an in-memory bucket) keeps the rate limit correct across restarts. Caps live in `db.py` per the D32 single-source precedent. **P1 reaches only the connected set**; durable announcements for late joiners (P2/BD6) are deferred. **Refines D6/D24; builds on D20/D30/D31.** | Reuse `kind="result"` for announcements (conflates "your task's outcome" with "a broadcast"; a distinct kind badges and reads cleanly); an in-memory rate-limit bucket (lost on restart — the abuse window reopens every boot); an allowlist / maintainer-only gate (heavier than the need; caps suffice for P1); per-recipient `enqueue_message` calls (N connections vs one multi-row transaction) |
| D34 | **Registry reads surface liveness-derived status from one shared source (AHB-15)** — `db.get_all_agents` returns `status` already derived by a new `db.derive_status(stored_status, last_seen, now, stale_threshold)`: explicit `offline` is preserved; anything else reads `online`/`stale` by `last_seen` age vs `STALE_THRESHOLD`. `/api/state` drops its inline duplicate of the same logic; the MCP `list_agents` tool (which previously returned the raw sticky column) inherits the derivation for free | The stored `status` column is only the sticky online/offline **intent** flag (D6/D23); actual presence is `last_seen` age. `list_agents` returning the raw column reported agents idle for 20+ days as `online`, misleading peers into routing tasks to dead agents (reported by `wiki-forge`, reproduced same-day by the maintainer). Deriving **inside the single accessor both surfaces share** makes divergence structurally impossible, rather than relying on each surface to remember to call a helper. Defensive: a `NULL last_seen` reads `stale` instead of crashing (the old `/api/state` inline code would have thrown). `db.broadcast`'s recipient selection deliberately stays on the **stored** column (BD3 targets online + stale, skipping only explicit offline) — unchanged | Fix only the `list_agents` tool with its own inline derivation (the exact duplication that caused the divergence); a helper both `hub.py` call-sites must remember to invoke (divergence still possible for the next consumer); write derived status back to the DB on read (mutating reads; the column would fight the sweep and the sticky intent semantics) |
| D35 | **Durable announcements via register-time catch-up with structural dedupe (AHB-1 P2)** — the P1 `broadcasts` audit table is the durable announcement store (gained the `context` column via a try/except `ALTER` migration); a new `db.deliver_missed_broadcasts(agent_id, window=BROADCAST_CATCHUP_WINDOW=24h)`, called by `register_agent` after the upsert, queues every in-window broadcast the registrant has **no message row for** as fresh pending `kind="announcement"` rows (`created_at=now`, so the D24 sweep gives them a full life; the register tool's return string notes the count). Dedupe is **structural**: P1 fan-out rows and catch-up rows both carry `session_id = broadcast_id`, so an existing row **in any status** (claimed, completed, expired) blocks re-delivery — no read-cursor column, idempotent across re-registers. ## Tunable Constants (proposed defaults — confirm)

| Constant | Default | Effect of changing |
|----------|---------|--------------------|
| `VISIBILITY_TIMEOUT` | `600s` | Time before an unacked `in_progress` message reverts to `pending`. Lower = faster redelivery after a crash, but more duplicates for genuinely slow tasks. **Raised 300→600s (Q5)** to fit agent-length work (Claude Code tasks routinely run >5 min) and cut false redeliveries; at-least-once means a too-low value only adds duplicates, not data loss. |
| `STALE_THRESHOLD` | `90s` | How long an agent can be silent before the dashboard marks it "stale". With long-poll, an on-duty agent re-calls `check_inbox` within its poll timeout and stays fresh. |
| `LONGPOLL_TIMEOUT` (default) | `30s` | Max time a single `check_inbox(wait=true)` call blocks before returning empty. Must stay safely under any client/proxy idle timeout. |
| `LONGPOLL_INTERVAL` | `~1s` | Poll cadence of the async long-poll loop (D21) — how often `check_inbox(wait=true)` re-checks the DB between `asyncio.sleep`s. Lower = snappier delivery + more idle queries; must stay well under `LONGPOLL_TIMEOUT`. |
| `DASHBOARD_MESSAGE_LIMIT` | `100` | Rows returned by `/api/state` (caps unbounded growth of the response). |
| `MESSAGE_TTL` | `86400s` (24h) | A `pending` message unclaimed for this long is swept to the terminal **`expired`** state (D6/Q3). Higher = friendlier to an agent that returns the next day; lower = more aggressive self-cleaning. |
| `RESTART_EXIT_CODE` | `42` | The sentinel exit code `POST /api/restart` exits with (`os._exit(42)`) and the only code the `run_hub.py` supervisor relaunches on (D26). Any other exit code stops the supervisor. A protocol constant, not a tuning knob — change it in **both** `hub.py` and `run_hub.py` together. |

### Competitive landscape & design-validation survey (2026-06-15)

Ran a four-track prior-art sweep (agentgateway; A2A + hermes-agent; MCP messaging servers; CLI orchestrators + agent frameworks), verified against primary repos/specs. Condensed below; fuller notes in `sessions.md`.

**Closest prior art (new — we weren't tracking these):**
- **[claude-peers-mcp](https://github.com/louislva/claude-peers-mcp)** (~2.1k★, active) — nearly our exact concept: a local SQLite broker for Claude Code peers (registers, polls every ~1s, pushes via the experimental `claude/channel`). **But Claude-Code-only** (stdio + `claude/channel`, won't reach Gemini/Antigravity) and **fire-and-forget — no durable queue/acks**. Our edges vs it: **CLI-agnostic** (Streamable HTTP) + **durable at-least-once queue**. Position against it.
- **[bobnet-mcp](https://github.com/cath42/bobnet-mcp)** — same idea, in-memory; *explicitly defers* persistence + future-delivery, i.e. punts on exactly the two hard features we build (confirms they're the differentiators).
- **[postal-mcp](https://github.com/tkellogg/postal-mcp)** (4★, a weekend experiment — finally read in full): two tools, a *blocking* `check_mailbox`, SQLite mailbox, **no ack/redelivery, no registry**. Its README confirms our core caution verbatim — a blocking receive *"doesn't return to the mailbox easily… takes a lot of prompting"* → **our documented work-loop is the real differentiator, not the long-poll itself.**

**What the field validates (keep — no change):**
- **Long-poll for pull-only clients** — postal-mcp and [hbd/mcp-chat](https://github.com/hbd/mcp-chat) both chose a blocking tool call for our exact reason ("works with ANY MCP client, no inbound port"). mcp-chat bounds it at 60s — matches our bounded `LONGPOLL_TIMEOUT=30s`.
- **Central registry + capabilities discovery** — every serious project has one; we're mainstream. **Durable SQLite store** — mainstream (CrewAI `SQLiteFlowPersistence`, LangGraph checkpointers); the fire-and-forget projects' main weakness is exactly the durability we add.
- **At-least-once + visibility-timeout** — we're a deliberate **outlier** (everyone else is ack-mailbox or no-ack); strictly more robust for "agent crashed mid-task." Justified.
- **Transport/trust model confirmed against the MCP spec** — single `/mcp` Streamable HTTP (replaces legacy SSE); a held-open SSE legitimately powers a blocking long-poll; the spec **mandates `Origin` validation + recommends binding `127.0.0.1`** — exactly our model. *Doc-currency note:* our docs cite spec rev `2025-03-26`; current stable is `2025-06-18`, and `2025-11-25` exists.

**Recorded as future options (not v1):**
- **A2A** is now an LF project (Google-donated, merged with IBM ACP), 1.0.x — borrowing its *vocabulary* keeps us cheaply interop-compatible. **hermes-agent's A2A support is proposal-only (unimplemented)**; its design intent (MCP-for-tools + A2A-for-peers, `INPUT_REQUIRED` for mid-task questions) is exactly what our hub brokers.
- **MCP `tasks` utility** (spec `2025-11-25`, experimental): `taskId` + `tasks/get` poll + TTL — mirrors our queue + visibility-timeout almost 1:1. Future interop path, not a v1 dependency.
- **MCP-UI / "MCP Apps" (SEP-1865, official Jan 2026)** is real but needs a graphical iframe host — **CLI agents (Claude Code, Gemini CLI) don't render `ui://`**. Keep the plain localhost dashboard (reinforces D8); rejecting MCP-UI is the right call.
- **agentgateway** (LF/Solo.io, 3.3k★) is a **stateless Rust proxy with no durable queue** — different category; nothing to adopt beyond prefix-namespacing as a discovery convention. Confirms rejecting A2A-as-transport (even its flagship gateway treats agent comms as sync routing).
- **[Agent Client Protocol](https://agentclientprotocol.com) (ACP, Zed Industries)** — evaluated 2026-06-16 (full note: `mem/acp-evaluation.md`). **Distinct from IBM's Agent _Communication_ Protocol above** — this is the "LSP for coding agents": a **vertical** editor↔agent protocol (JSON-RPC over **stdio**, the editor spawns the agent as a subprocess; Gemini CLI is the reference agent, Claude Code ships via an adapter). **Rejected as transport** for the same reason as A2A but sharper — wrong topology (editor↔agent, not agent↔agent; no peer/registry/inbox concept), and using it would force the hub to masquerade as an "editor" spawning each agent. **But it layers cleanly *above* us:** an ACP agent is told which MCP servers to use via `session/new`'s `mcpServers` (http supported), so an ACP-hosted Claude Code / Gemini CLI reaches the hub **unchanged** via the editor's http `mcpServers` — **validating D1**. Also a **4th converging validation of D17** (`session/request_permission` = pause-and-ask-back-then-resume; A2A `input-required` / LangGraph / CrewAI were the first three; ACP's *typed `outcome` enum* is a v2 idea for a clarification cancel/reject path). Positioning: the hub is the **horizontal, durable, local peer complement** to ACP's vertical editor↔agent standard.
